Root Cause:

The SendMessages() function assumes that CNodeState always exists for a connected peer. It dereferences the result of State() without checking for NULL. While State() can return NULL if the NodeId is not found in mapNodeState, the code directly dereferences it as a reference:

```cpp
CNodeState &state = *State(pto->GetId());
```

If State() returns NULL, dereferencing NULL causes undefined behavior, typically a segmentation fault and node crash.

Differential (Not a consensus divergence, but DoS vector):

Node behavior with State entry: Processes messages normally
Node behavior without State entry: Crashes with NULL dereference

Divergence trigger: 
- Race condition where FinalizeNode() completes (removing mapNodeState entry)
- But SendMessages() is still invoked for the same NodeId
- Timing window between node disconnection and message handler cleanup

Exploit Scenario:

Step 1: Attacker establishes connection to victim node
Step 2: Attacker triggers rapid disconnect/reconnect
Step 3: Race condition: SendMessages() called after FinalizeNode() removes state
Step 4: Victim node crashes with NULL pointer dereference
Step 5: Victim node requires manual restart
Step 6: Attacker repeats to cause persistent DoS

Required privileges: NONE (any P2P peer can trigger)
Required network position: ANY (direct connection to victim)

- Type: Remote Denial of Service (node crash)

- Recovery: Manual restart required

Affected node population:
